## 📋 风险登记册明细

### R-01：表格输入异常数据与非法值入库风险
* **风险描述**：用户在高频录入中无意输入了非数字字符（字母、符号）、负数值或极端极大值，由于前端表格未作硬性过滤防呆，垃圾数据成功写入 Firestore，导致下游 `CalculationEngine` 等计算引擎在运算浮点数时出现 `NaN`、`Infinity` 或直接爆栈崩溃。
* **严重度 (Severity)**：⭐⭐⭐⭐ (4/5, 高危)
* **可能性 (Likelihood)**：⭐⭐⭐⭐ (4/5, 高发)
* **缓解对策 (Mitigation)**：
  - **失焦防守线**：单元格绑定严格的数据校验器，在失焦失焦（Blur）瞬间，若为非法输入则整表标红，或立即强行回弹重置至输入前数值。
  - **动作阻断**：只要表格存在一处非法输入红线，全局 [保存 (Save)] 按钮实施物理禁用，严防非法数据入库。
* **Owner**：CC 开发团队负责人

---

### R-02：Firestore 密集写入放大与企业计费暴增风险
* **风险描述**：由于销售预测（Forecasts）数据呈高频、高维录入特征，若开发团队采用失焦即存（Auto-save on blur）或不当的 debounce 自动保存，频繁写入将触发 Firestore 计费爆炸；同时短时间并发大量 set 会超出单文档每秒 1 次的物理限制，导致数据库写入卡死。
* **严重度 (Severity)**：⭐⭐⭐⭐⭐ (5/5, 灾难级)
* **可能性 (Likelihood)**：⭐⭐⭐⭐ (4/5, 高发)
* **缓解对策 (Mitigation)**：
  - **强制阻断 Auto-save**：死守 Dirty State (Save/Discard) 交互模型，所有修改存入 React 内部内存快照中。
  - **批量合并 Patch 写入**：点击保存时，前端做精确的 Patch 对比，仅打包有实际变化的文档，通过 `writeBatch()` 进行打包单次 IO 物理写入，计费降低 95% 以上。
* **Owner**：系统架构师

---

### R-03：Workspace Viewer 只读隔离前端漏防越权篡改风险
* **风险描述**：在进行 react-datasheet-grid 引入及视觉 Parity 改造时，重写样式如果仅在隐藏了 [Save] 按钮，但未将表格底层数据网格的 `disabled` / `readOnly` 物理属性设为 true。这会导致只读（Viewer）用户双击依然可以进入编辑状态并修改本地状态，产生“协同权限已篡改”的虚假安全感与严重权限漏防。
* **严重度 (Severity)**：⭐⭐⭐⭐⭐ (5/5, 灾难级)
* **可能性 (Likelihood)**：⭐⭐ (2/5, 较低)
* **缓解对策 (Mitigation)**：
  - **DOM 级硬性防守**：在 react-datasheet-grid 组件外层或 Column 参数中，强绑定 `readOnly: UserRole === 'Viewer'`。
  - **后端安全规则硬拦截**：确保 `firestore.rules` 对只读角色依然保持最高的后端写拒绝机制（本轮不改 rules，但前端需配合硬防守）。
* **Owner**：QA 负责人 & CC 安全开发员

### R-04：多币别与 BP 目标折算精度脱节及对账不齐风险
* **风险描述**：年度 BP 目标单位为 Million TWD，而销售 SKU 预测包含 USD、CNY 及 TWD。如果汇率折算公式混乱，未对齐时间切片，会导致两端对账时出现微小的汇率截断偏差，导致高管与业务人员在对账单上产生疑惑和不信任。
* **严重度 (Severity)**：⭐⭐⭐⭐ (4/5, 高危)
* **可能性 (Likelihood)**：⭐⭐ (2/5, 较低)
* **缓解对策 (Mitigation)**：
  - **美元中转折算铁律**：全产品销售折算公式死守统一标准化两步法：`销售预测多币种 -> normalize 为 USD -> 转为 TWD 与 BP Targets 比较`。
  - **时间切片同步**：折算时引入系统中定义的实时及历史固定汇率常数 `usdToTwd`，防止因时点漂移导致两边取值不同。
* **Owner**：核心财务计算引擎开发员

---

### R-05：AI Export 文本分析结论被盲目迷信为决策自动化风险
* **风险描述**：高管用户在使用 AI 导出（AI Brief Export）时，由于 AI 生成的报告文字流畅、论据看似充分，用户盲目迷信其推荐的裁撤 SKU 或调整产能决定，未做二次财务校验直接拍板，引发商业决策偏失。
* **严重度 (Severity)**：⭐⭐⭐ (3/5, 中等)
* **可能性 (Likelihood)**：⭐⭐⭐⭐ (4/5, 高发)
* **缓解对策 (Mitigation)**：
  - **强加安全免责出口**：在导出的 AI 报告头部和尾部，强行加入优雅但显眼的免责警示水印：“*注：本报告为数学比例分摊分析，不构成直接的投资因果因果判断。请结合物理产能与财务团队进行二次确认。*”
  - **措辞治理**：将 AI 分析生成的 Prompt 中的 “Causality (因果)” 指向彻底治理为 “Attribution (占比归因)”。
* **Owner**：AI 功能产品经理 (PM)

---

### R-06：Snapshot Immutable 强只读被用户误解的风险
* **风险描述**：用户在 Snapshot History 列表中，可能会误以为生成的快照（Snapshot）可以被二次编辑和更新（混淆了工作版 Work Version 和只读快照的历史版本）。这会导致用户反复尝试修改无效，引发对系统数据保存可用性的质疑。
* **严重度 (Severity)**：⭐⭐⭐ (3/5, 中等)
* **可能性 (Likelihood)**：⭐⭐⭐⭐ (4/5, 高发)
* **缓解对策 (Mitigation)**：
  - **视觉锚定只读态**：只要用户切换进入任何历史 Snapshot 视图，页面标题、头部背景及全局边框必须转变成温和的“快照只读专有橙灰色”，并在页头明确浮现“快照已归档 immutable 只读”的水印提示。
  - **拦截动作卡片**：在快照查看态下，所有的 [Save] 按钮被优雅替换为 [复制此快照作为最新工作版 (Clone to Work)] 动作，化阻断为有益的指引。
* **Owner**：UX/UI 设计师
