# v1.31 Dashboard & Results UI 重構手動冒煙測試腳本

本文檔為 `v1.31.0` 的手動冒煙測試（Smoke Test）腳本，供 QA 人員或開發者在重構完成後，在瀏覽器端進行直觀的視覺與交互驗證。

---

## 測試環境準備
1. 確保本地開發伺服器正常運行：`npm run dev`。
2. 啟動瀏覽器並打開系統（預設：`http://localhost:5173`）。
3. 準備兩個測試帳號：
   - **帳號 A (Workspace Owner 或 Editor)**：具備完整寫入權限。
   - **帳號 B (Workspace Viewer)**：僅具備唯讀權限。

---

## 🧪 測試案例 1：儀表板 (Dashboard) 視覺與佈局測試

### 測試步驟
1. 使用 **帳號 A** 登入系統。
2. 點擊左側導航欄「Dashboard / 儀表板」。
3. 觀察頁面整體佈局。
4. 將瀏覽器視窗寬度拉窄至 `375px`（手機視口模式）。

### 預期結果
- [ ] 頁面頂部顯示標準的 `<PageHeader>` 區塊，標題與描述清晰，字體對齊一致。
- [ ] 頁面整體四周擁有標準的 `.abf-page` 邊距內距，與 BpTargets 頁面完全對齊。
- [ ] 頂部 6 個 KPI 卡片、BP卡片與圖表區卡片，其圓角（6px）與標題樣式統一為 `SectionCard` 小尺寸風格。
- [ ] 所有產能數據旁邊的單位顯示為灰字 `(panels/day)`，並完美對齊數值。
- [ ] 底部所有備註文字（如數據限制說明）呈現為 12px 灰色斜體。
- [ ] **RWD 測試**：在 `375px` 窄屏下，6 個 KPI Card 自動轉為單列垂直堆疊排列，圖表卡片自適應寬度，無任何橫向溢出或欄位擠壓變形。

## 🧪 測試案例 2：計算結果 (Results) KPI 與 Toolbar 測試

### 測試步驟
1. 點擊左側導航欄「Results / 計算結果」。
2. 觀察頁面頂部 Summary KPIs 區域。
3. 將瀏覽器視窗寬度拉窄至 `375px`（手機視口模式）。
4. 觀察 View 視圖切換控件與 Snapshot Compare 區域。

### 預期結果
- [ ] Results 頁面頂部 Summary KPIs 與 Dashboard 的 KPI 卡片風格、大小及字型完全一致。
- [ ] **RWD 測試**：在窄屏下，Summary KPIs 自動垂直折行，無橫向溢出。
- [ ] 頂部 View 視圖切換的 `<Segmented>` 控件及周邊操作元件包裹於 `.abf-toolbar` 中，背景為純白，有微小的邊框線。
- [ ] 窄屏下，Toolbar 內的按鈕群完美 wrap 自動換行，無超出版面寬度。

## 🧪 測試案例 3：寬表水平滾動條 (不擠壓寬表) 測試

### 測試步驟
1. 在 Results 頁面中，點擊視圖切換為「Sales View」或「Product Planning View」。
2. 觀察 `TimeMatrixTable`（包含月度/年度橫向展開寬表）的呈現。
3. 將瀏覽器視窗寬度拉窄至 `768px`（平板視口）與 `375px`（手機視口）。
4. 嘗試向右拖動表格。

### 預期結果
- [ ] 在平板與手機窄屏下，整個頁面寬度**沒有**被撐大，網頁沒有產生全網頁的橫向滾動條。
- [ ] `TimeMatrixTable` 內部產生了流暢的局部橫向滾動條，滾動條完美貼合卡片邊緣。
- [ ] 表格內部的所有數據欄位寬度固定（無擠壓變形），價格、數量與產能數據清晰可讀。

## 🧪 測試案例 4：只讀模式 (Viewer) UX 與安全測試

### 測試步驟
1. 登出 **帳號 A**。
2. 使用 **帳號 B (Workspace Viewer)** 登入系統。
3. 點擊「Results / 計算結果」，並切換視圖至「Change Review」。
4. 觀察頁面頂部與「新建版本」按鈕。

### 預期結果
- [ ] Results 頁面頂部（或 Change Review 區塊頂部）高亮顯示標準的藍色唯讀警告：*「您在此工作區為 Viewer 角色。編輯功能已被禁用。」*
- [ ] 「新建版本」按鈕呈 `disabled` 灰色狀態，鼠標移上去無法點擊。
- [ ] 點擊「匯出 JSON」或「下載 JSON」依然正常運作，只讀數據正常加載。

---

## 🧪 測試案例 5：展示層 NaN 與空值防護測試

### 測試步驟
1. 進入一個包含空白或異常預測數據的專案。
2. 切換到 Results 頁面，觀察「瓶頸分析」與「利用率分析」。
3. 檢查是否有分母為 0 或數據缺失導致的利用率欄位。
4. 切換多國語言為「English」與「繁體中文」，觀察所有翻譯。

### 預期結果
- [ ] 任何百分比或利用率欄位，在數據缺失時**優雅顯示為 `-`**，在畫面上**絕對不能**看見字面 `NaN%` 或 `NaN` 或 `Infinity`。
- [ ] 貨幣金額根據 prefs 設定在切換時自動換算，無硬編碼符號錯誤。
- [ ] 切換為「English」時，整個 Results 頁面（包含 Risk Brief, Change Review, AI Export 區域的 notice、disclaimer、鈕扣文字）100% 變為英文。
- [ ] 切換為「繁體中文」時，所有翻譯精確，無 mojibake 亂碼，無硬編碼英文。
