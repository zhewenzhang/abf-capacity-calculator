# Phase 6.1 Forecast Snapshot Change Review 体验验收卡

本文件定义了 **ABF Capacity Calculator** 在 **Phase 6.1 Forecast Versioning & Change Impact Review**（变动比较与影响审查大盘）二次体验打磨迭代中的 **UX 质量与安全验收标准**。本验收卡旨在为协同开发团队 (CC) 后续针对比较大盘的交互优化（UX Polish）提供精确、硬性的质量交付基石。

---

## 一、 核心产品目标与用户故事

### 🎯 产品目标
当输入数据（Forecasts, skus 等）发生版本更迭后，能够通过极致清晰、具备专业决策美感的 UI 交互，直观地帮助决策者对两个历史快照（Snapshots）之间的指标波动进行**量化比对与 attributions（比例分摊）归因**，彻底辅助其锁定营收和产能波动的宏观驱动力，同时保证离线脱敏数据的强安全导出。

### 👥 核心使用者故事 (User Stories)
- **Sales (业务)**: “我需要比对调整前后的快照，量化看出这次微调中是哪个客户或 SKU 占据了营收 Delta 的主要比例，且其驱动力是价格还是数量，以此作为与客户追加单价的谈判筹码。”
- **Capacity Planner (工厂产能)**: “我需要一眼比对出两个预测版本带来的瓶颈短缺月份（Shortage Months）增减、Core/BU Max Utilization 波动，以判定是否需要调整新机台的采购计划。”
- **Executive (决策层)**: “我需要秒开一个最干净的 Delta 差异面板，掌握全年 BP Attainment 达成率是升高还是降低了多少个百分点，拒绝繁琐的底层数据，只看核心 KPI Delta。”

## 二、 Snapshot List (快照列表) 体验验收标准

快照列表位于用户管理版本存档的入口，其验收要求如下：

1. **基本字段完整度**：每一条快照条目必须包含以下基本要素：
   - 快照名称 (Name) 与 快照描述 (Description) — 必须支持中文显示，溢出时应有 Tooltip 展开提示。
   - 创建人 (CreatedBy) — 必须包含创建者名字，便于在多用户共享工作区中定位。
   - 创建日期 (CreatedAt) — 必须格式化为本地易读格式（如 `YYYY-MM-DD HH:mm`）。
   - 源系统版本 (Source App Version) — 必须清晰标注创建时的应用版本号（例如 `v1.22.2`），用以向后追溯计算物理模型。
2. **Viewer 角色硬隔离**：
   - 工作区的 Viewer 角色在打开快照列表时，**必须被物理隐藏或禁用「创建快照」与「删除快照」按钮**。
   - Viewer 鼠标移过相关被禁用的图标时，必须有气泡明确告知“您当前属于只读访客角色，无权变更快照”。
3. **性能体验**：
   - 快照列表加载时需配备优雅的 Skeleton 或 Spin 动画。
   - 支持根据创建时间 `createdAt` 降序排列（最新快照自动置顶）。

## 三、 Compare Selector (对比选择器) 交互验收标准

选择器用于确定哪些快照用于比对，其方向性是决策大盘的灵魂。

1. **明确的 Base 与 Compare 区分**：
   - UI 上必须极其清晰地向用户标记：**Base Snapshot (比较基准)** 与 **Compare Snapshot (比较目标)**。
   - 建议在选择器上方或侧边有明确的信息图标（Tooltip）阐明物理意义：
     - *Base (基准)*：表示调整前（旧版）的历史底档。
     - *Compare (比较)*：表示微调后（新版）的最新数据。
2. ** Compare - Base 计算方向硬性一致性**：
   - 决策大盘中所有的 Delta 差值（无论是百分比还是绝对值），**必须严格、一致地按照 `Compare (新版) - Base (旧版)` 这一方向进行计算**！
   - *例如*：如果 Base 营收为 1000 万，Compare 营收为 1200 万，则 `Delta` 必须为 `+200万` 且显示为代表上升的绿色；严禁颠倒计算为 `-200万`，彻底防范方向性决策误导。
3. **单项目与同工作区约束**：
   - 在选择器中，只允许列出**当前活跃项目和工作区下**的快照列表。严禁跨项目或跨工作区进行快照交叉对比。
   - 当快照总数不足 2 条时，选择器应默认处于禁用状态，并友好提示“当前快照不足 2 条，请先去各页面保存新的快照底档”。

## 四、 Revenue / BP / Capacity / Top Changes 模块验收标准

指标比较大盘是用户的核心决策战场，信息层级要求极致严密：

1. **核心大盘 KPI 指标 Delta (Facts 事实区)**：
   - 必须配备宏观差异卡片，直接展示：
     - **Revenue Delta** (USD / TWD / CNY) — 绝对差值与百分比。
     - **BP Attainment Delta** (百分比差值) 与 **BP Gap Delta** (Million TWD 绝对值差)。
     - **Shortage Month Delta** (短缺月份数差异，如 -2 个月) 与 **Worst Bottleneck Month** 位移情况。
     - **Max Core / BU Utilization Delta** (最大产能利用率波动百分点)。
2. **Attribution (比例归因) 区域与安全拦截**：
   - **比例分摊的强定义**：大盘中展示的“变动最大的 SKU”、“变动最大的客户”、“变动最大的月份”等 Attributions 归因表，**其标题和表头必须有明确的“比例分摊 / Proportional”或“Attribution”字样**。
   - **安全警示文案要求**：该区域侧边或底部必须配备常驻的、醒目的安全 Guardrails 警示：
     > ⚠️ [安全声明] 此处的变动项目排名仅表示该项目在营收/预测的 Delta 差值中**所占的账面比例最大，属于比例分摊，并不表示该项目是导致整体变动的因果源头**。严禁将比例分摊错认作 causal（因果）责任模型。
3. **价格 vs 数量驱动分析 (Attribution) 的物理限制**：
   - 大盘中提供的 Price-driven (价格驱动) 与 Quantity-driven (数量驱动) 的营收 Delta 占比计算，必须标明为“一级一阶物理效应估算（First-order effects）”。
   - 严禁声称此分析是“多变量因果联动因果模型”。
4. **多币别与 BP Attainment 的完美对齐**：
   - 当用户在 App 顶部切换显示币别（USD / TWD / CNY）时，变动大盘中的所有 Revenue Delta 数值**必须同步、完美地根据 parameters 中设定的换算汇率进行更新渲染**！
   - **BP Gap 必须强制换算并展示为 `Million TWD` (百万台币)**，与系统全局的 BP Targets 单位标准保持绝对一致，严禁出现币别混淆。

## 五、 Empty States (空状态) 体验验收标准

1. **快照列表为空时**：
   - 列表应展示 AntD Standard Empty 状态。
   - 应有友好的新手引导文案，清晰指导用户“前往 Forecasts 预测页、Products 产品页或 Capacity 产能页，编辑好数据后在侧边栏/设置中建立您的第一个快照存档”。
2. **比较大盘未选择快照时**：
   - 大盘主区域不应直接抛错或显示空白，必须展示“请在上方选择器中指定您的 Base 基准快照与 Compare 目标快照以开启决策比对”的插画与引导卡。
3. **两快照无任何 Delta 差异时**：
   - 如果用户选择了两个数据完全一致的快照进行比对，Delta 卡片应正确显示 `0`，Attribution 列表应展示“两版本数据无变化”的清爽 Empty 状态，而不是渲染出一堆 `0%` 的空白列表。

## 六、 AI Export (DeepSeek 导出包) 安全验收标准

1. **AI 导出的强脱敏物理边界**：
   - 导出的 Change Impact Pack，**其底层 JSON 数据必须 100% 遵守 sanitization 脫敏策略**（剥离个人敏感 UID、模糊组织与敏感名称），仅暴露出与变动比例和数字 Delta 相关的分析底料。
   - 严禁在导出文本中包含未经脱敏的用户个人或工作区敏感标识。
2. **AI Guardrails 警示锁死**：
   - 导出的 Combined Pack（Prompt + JSON）中，**提示词 (Prompt) 必须百分之百包含 F-A-I-R (事实/归因/推论/建议) 的标记前缀要求**。
   - 提示词中必须以**一票否决式的高压口径**，强行锁死“禁止大模型做出商业决策”、“禁止大模型擅自篡改物理计算结果”、“禁止大模型将比例分摊解读为因果责任分配”的安全红线。
3. **本地 JSON 下载的 Excel 友好性**：
   - 点击“下载 Change Impact JSON 文件”导出的文件，**其内容开头必须物理写入 UTF-8 BOM 字符 (`\uFEFF`)**。
   - 这能保证用户将导出的 JSON 文件直接用 Microsoft Excel 或其他表格软件双击打开时，**绝不发生中文字符乱码（mojibake）**，极大提升决策体验。

## 七、 体验硬性“反面”红线 (Out of Scope / Anti-patterns)

在验收时，一旦 CC 的交付包含以下任一现象，必须直接予以驳回重做：
- ❌ **任何形式的 AI API 直接接入**：本项目定位为 100% 确定性本地沙盘。比较大盘内绝不允许接入任何联机 AI API、不允许内置大模型聊天框，所有 AI 辅助必须保持“手动复制 sanitization 数据包至外部 AI 软件”的旁路物理隔离模式。
- ❌ **修改任何 core 物理计算公式**：变动大盘仅用于“读取”两个快照 rawInputs 并进行差值比对，**绝不允许在比对层修改系统既有的 core 产能、营收、BP 等物理计算公式**，防范公式漂移风险。
- ❌ **大型花哨 UI 重构**：比较大盘必须沿用项目既有的 Ant Design 6 设计语言，卡片、表格样式应与 Dashboard/Results 完美一致，坚决反对应付差事的、破坏一致性的大型花哨重新设计。
